# Maintenance Loop

Use this reference when a user wants ongoing help after the initial structure exists. The goal is to keep knowledge useful, current, and fast without losing source depth.

## Add New Knowledge

1. Identify the file type: source, index, workflow, checklist, output, identity/rule, or archive material.
2. Place source-of-truth material in the source shelf or the user's equivalent canonical area.
3. Preserve provenance: original filename, source URL, original path, author, date, or import note when available.
4. Update the relevant index, key-links file, recurring-docs file, README route, or loader.
5. Verify the new file is reachable without adding it to default startup context unless it is truly startup-critical.

## Refresh Existing Knowledge

- Re-read the full source before editing an index or summary.
- Treat indexes as maps, not replacements.
- Prefer dated additive notes for updates when history matters.
- Keep exact values, policies, definitions, and compliance details in canonical sources.
- Archive superseded drafts or generated outputs instead of mixing them into sources.

## Periodic Health Check

- Broken relative links.
- Duplicate filenames or competing source-of-truth files.
- Unreferenced source-like files.
- Startup files growing beyond their intended size.
- Indexes that no longer mention new or renamed sources.
- Archive and manifest coverage for recent migrations.

## Maintenance Report Template

```text
Knowledge base health:
- Sources preserved: yes/no
- New or changed files routed: yes/no
- Indexes refreshed: yes/no
- Broken links found: N
- Unreferenced source-like files: N
- Startup load: X words, about Y tokens

Recommended next action:
- One small action the user can approve or perform next.
```
